垂直居中有哪些实现方式

块级元素：

  • 绝对定位 + top: 50% + margin: -50%高度

    需要知道子元素宽高，父元素position设置为除static以外的值

  • 绝对定位 + top: 50% + transform：translate(0, -50%)

    不需要知道子元素宽高，父元素position设置为除static以外的值

  • 绝对定位 + top/left/bottom/right : 0 + margin：auto

    父元素position设置为除static以外的值，top、left、bottom、right都要为0

  • 相对定位 + top: 50% + transform：translate(0, -50%)

    优点：不需要更改父元素，但是元素不会脱离文档流。

  • flex布局 / grid布局，设置align-items: center

  • table布局，设置父元素为display: table-cell，子元素设置 display: inline-block。利用vertical和text-align可以让所有的行内块级元素水平垂直居中

行内元素：

  • 设置line-height和元素高度相同，可以实现单行文本垂直居中
  • 设置line-height和元素高度相同、vertical-align: middle可以实现行内元素居中

position: absolute，relative，fixed

  • position: absolute代表绝对定位，设置绝对定位的元素会从文档流中移除，通过top、left、bottom、right属性确定位置。定位的上下文元素是其最近的position属性不是默认值的父元素，一般设置为relative。如果所有父元素的positon属性都时默认值，就相对于视口定位。
  • position: relative代表相对定位，它不会影响元素的布局，设置top、left、bottom、right属性相对于原本位置偏移，偏移量不会影响其他元素。
  • position: fixed代表固定定位，行为和绝对定位类似，区别是它的定位上下文是视口

圣杯布局：

  1. 左中右三个元素设置左浮动
  2. 左右元素定宽，中间元素宽度100%
  3. 左元素设置margin：-100%，使得左元素上升一行并处于最左侧；右元素设置marigin自身宽度的负值，使得右元素上升一行并处在最右侧。
  4. 设置父元素的左右padding为左右元素预留空间。
  5. 设置左右元素为相对定位，左元素的left和右元素的right为内边距的负值

双飞翼布局：

  1. 左中右三个元素设置左浮动
  2. 左右元素定宽，中间元素宽度100%
  3. 左元素设置margin：-100%，使得左元素上升一行并处于最左侧；右元素设置marigin自身宽度的负值，使得右元素上升一行并处在最右侧。
  4. 中间元素设置一个子元素，设置子元素的左右边距，给左右布局预留空间。

双飞翼布局和圣杯布局的前三步是一样的，区别在于展示内容布局的方式。圣杯布局采用父元素设置padding，左右布局设置相对定位的方式；双飞翼布局则使用嵌套子元素设置外边距的方式。

  • 封装：将属性和方法封装在对象内部，并且控制成员的访问权限。可以提高内聚性，降低耦合性，便于维护。

this的指向

this的指向取决于上下文，上下文包括函数上下文、全局上下文以及类上下文。

  • 对于全局上下文，this指向全局对象，由运行环境决定，一般是window。
  • 对于函数上下文，如果函数直接被调用，例如回调函数，this为全局对象window（严格模式为undefined）；如果是由对象调用的函数，this就是这个对象；如果是构造函数，this为正在构造的对象；对于箭头函数，this的指向等同于创建函数时所处上下文的this，并且保持不变不能更改。
  • 对于类上下文，如果是实例字段，this指向对象实例；如果是静态字段，this指向类本身。
  • 可以通过call apply bind方法显示设置函数的this值。

什么是闭包？有什么应用场景？

  • 什么是闭包？

    闭包是指能访问外部函数作用域的函数，内部函数存在外部作用域的引用就会形成闭包。闭包的作用是创建私有作用域，保存函数的私有变量

  • 闭包的使用场景

    • 防抖、节流。以及其他需要定义函数私有变量的情况。
    • 循环赋值。利用闭包会形成私有作用域的特点。

  • 闭包变量在哪里存储（？）

    堆内存。假如闭包中的变量存储在栈内存中，函数执行完后，栈帧被销毁，闭包变量也就跟着销毁了。

  • 使用闭包要注意什么？

    内存泄漏。如果一个变量持有内部函数的引用，内部函数又持有外部函数的引用，就发生了内存泄漏，外部引用不会被回收。注意使用完闭包函数后，把指向内部函数引用的变量置空即可

原型、原型链

  • 什么是原型、原型链？

    原型是指构造函数的prototype属性，使用new构造对象时，会把这个原型对象赋值给实例对象的proto属性。它的作用是储存实例对象的公共属性，起到复用、节省内存的效果。

    而prototype原型对象也是一个对象，也具有_ proto _属性，因而构成了原型链。

    JS通过原型链实现继承。

  • JS对象类型

    JS对象分为函数对象和普通对象，只有函数对象有prototype属性。

  • Object、Function都是js内置的函数

    重点：

    • Object.proto = Function.prototype（因为Obejct是函数）
    • Function.prototype.proto = Object.prototype（因为原型对象prototype也是对象，所有对象的_proto_最终都会指向一号对象Object.prototype）
    • Object.prototype.proto = null

  • 属性屏蔽、属性覆盖

    实例对象重写原型上继承的属性、方法，相当于“属性覆盖、属性屏蔽”，这一操作不会改变原型上的属性、方法，自然也不会改变由统一构造函数创建的其他实例。

    只有修改原型对象上的属性、方法，才能改变其他实例通过原型链获得的属性、方法。

TS相关

什么是TS

TypeScript是JavaScript的超集，在JavaScript基础上添加了静态类型，提供编译时类型检查、类型推导等功能。能帮助开发人员规避一些常见的类型错误，提高代码的可读性和可维护性。

JS和TS的区别

JS是动态类型语言，不需要显示声明变量类型，变量类型在运行时确定，且可以改变。TS是静态类型语言，变量在声明时需要确定类型，并且提供类型检查、类型推导等功能，提高代码的可读性和可维护性。

什么是类型声明和类型推断？

类型声明是指在声明变量时，显示指定变量的类型；类型推断是指TS根据赋值语句右侧表达式的值自动推断变量的类型。

什么是接口（interface）？接口和类型别名的区别

接口是用来描述对象结构的结构化类型。它定义了对象应该包含哪些属性和方法。

而类型别名的作用是给类型起一个新名字，便于在多个地方复用。常见的使用场景是联合类型。

什么是泛型？

泛型就是类型参数，可以定义泛型类或者泛型函数。比如Set和Map类，其内部元素的类型不固定，在声明时指定，这种情况下就可以使用泛型。

什么是联合类型和交叉类型

联合类型表示一个值可以是多个类型中的一种；交叉类型表示一个值包括多个类型的特性。

响应式原理，vue2 和 vue3 响应式原理的区别

vue的响应式实现分为三步：

  1. 监听变量的访问和更新，也就是get和set方法
  2. 变量被访问时，将依赖项封装为Watcher实例收集起来
  3. 变量被更新时，通知依赖变量的所有Watcher更新

  • （vue2 和 vue3 响应式原理的区别）对于变量访问和更新的监听，也就是get和set方法的监听，vue2使用defineProperty()方法实现，vue3使用proxy代理对象实现。

    defineProperty()监听的是变量，对象的每个属性都要调用这个方法，深层对象需要递归调用。缺陷是不能监听属性增加和删除，不能监听数组的变化。

    proxy是ES6的新特性解决了defineProperty()的通点，并且由于proxy是针对整个对象的，只需要调用一次即可，但是深层对象还是需要递归调用。

  • 下一步是收集依赖。依赖项本质上就是一个表达式或者说一个函数，当变量更新时让这个表达式或函数重新执行一遍即可更新依赖项。vue定义了Watcher类来保存这个函数，get方法被触发时，说明变量被访问，这时收集一个封装好的Watcher实例即可。

  • 有了依赖项，还需要一个地方来储存依赖项，它就是Dep类，他负责保存watcher实例和触发watcher更新。当set方法被触发时，说明变量更新了，我们需要通知依赖项更新，只需要在Dep中找到对应的Watcher实例，执行回调即可。

Vue的响应式是通过监听变量的访问和更新，也就是get和set方法实现的。

当变量被访问时，将依赖项封装为一个Watcher实例并收集起来；当变量被更新时，通知所有依赖这个变量的Watcher执行回调，这些回调会通过nexttick发送到微任务队列，作为微任务执行。

computed和watch区别

  • 功能上：computed是根据多个属性计算一个属性的值，watch是监听一个值的变化并执行回调
  • 使用场景上：computed常用场景是简化template模板中的复杂表达式，模板中不应该存在太多逻辑判断，会导致维护困难；watch的常用场景是状态变化后执行一些异步操作。
  • 如何选择：当我需要一个计算属性时，选择computed；当我需要一个监听器时，使用watch。

vue2：

  • 首先调用 patch 判断两个虚拟dom的根节点vnode是不是都存在：

    • 如果旧节点oldVnode存在，新节点vnode不存在，删除旧节点；

    • 如果旧节点不存在，新节点存在，创建新节点；

    • 对于都存在的情况：

      如果是同一节点（key、tag都要相同），则调用patchVnode对比子节点和节点文本；

      如果不是同一节点，删除旧节点，创建新节点。

  • patchVnode 首先判断两个vnode的是否都有子节点：

    • 如果oldVnode有子节点，vnode没有子节点，删除掉所有子节点；
    • 如果oldVnode没有子节点，vnode有子节点，创建所有子节点；
    • 如果都存在子节点，调用updateChildren对比、更新新旧子节点（重点）

  • updateChildren会分别为新旧子节点数组创建头尾指针，然后while循环遍历直到一个列表的 startIndex>endIndex。每次循环依次进行如下流程：

    • 新的头和旧的头对比，如果是相同节点，则递归调用patchVnode对比更新

    • 新的尾和旧的尾对比，如果是相同节点，则递归调用patchVnode对比更新

    • 新的头和旧的尾对比，如果是相同节点，则递归调用patchVnode对比更新，并且将旧尾节点插入到旧头节点之前（保证相对顺序，和新子节点保持一致）

    • 新的尾和旧的头对比，如果是相同节点，则递归调用patchVnode对比更新，并且将旧头节点插入到旧尾结点之后（保证相对顺序，和新子节点保持一致）

    • 如果上面的判断都未命中，则给旧子数组构建一个key -> index的map，然后查找newCh[newStartIndex]（即新子节点数组的头指针指向的节点）在旧子数组中的索引：

      如果找到了，则递归调用patchVnode对比更新，并且将旧子数组的该节点插入到旧头结点之前（保证相对顺序，和新子节点保持一致）。

  • 循环结束

    • 如果oldStartIndex>oldEndIndex，说明老列表先遍历完，添加从newStartIndex到newEndIndex的节点。
    • 如果newStartIndex>newEndIndex，说明新列表先遍历完，删除从oldStartIndex到oldEndIndex的节点。

vue2和vue3的区别

  1. vue2使用选项式api，而vue3使用组合式api。

    选项式api容易上手，但是逻辑散乱，因为data、method、computed需要分开定义

    组合式api自由度更高，代码可读性、内聚性更强

  2. 响应式实现原理不同

    vue2对于get和set方法的监听，通过defineProperty()实现，它的第一个缺点是监听不到属性的新增和删除；第二个缺点是不能监听数组的变化；第三个缺点是因为defineProperty()是针对属性的，需要给对象的每个属性调用一遍，当属性过多时可能存在性能问题。

    vue3基于proxy实现对象代理，解决了defineProperty()的缺点。它支持属性增删的监听，支持数组代理，同时对于同级属性，做一次代理即可，不需要重复调用。

  3. diff算法的实现不同

    vue2首先会进行头头、尾尾、头尾、尾头比较，命中直接复用，都没命中就去旧节点列表遍历，命中就复用，还没命中说明是新节点，就新建一个节点。

    vue3首先进行头头、尾尾比较。对于剩余的节点，如果只有新节点有剩余，就直接创建；如果只有旧节点有剩余，就直接删除。如果新旧都有剩余，那就先拿到新节点在旧节点数组中的下标，得到一个下标数组，再找到这个下标数组的最长递增子序列，在这个序列里的节点直接复用，剩余的节点进行移动/插入/删除。

    使用最长递增子序列可以最大程度的减少 DOM 的操作次数。

Get和Post请求的区别

  • 定义和规范上的区别（本质区别），GET 请求是用来获取资源的，也就是进行查询操作的；而 POST 请求是用来修改服务器资源的，也是就用来进行增删改操作的。
  • 请求参数：get请求的参数暴露在url中，长度有限制，存在安全隐患；post请求参数存储在请求体中，没有长度限制，相对安全
  • 缓存：get请求默认会被缓存，post请求默认不缓存，除非响应头包含缓存相关字段例如Expires、Cache-Control
  • 历史记录不同：get请求的参数会保存在历史记录，post请求则不会。

tcp和udp的区别

  • tcp是面向连接的协议，udp是无连接的
  • tcp提供可靠的数据传输，udp不保证数据传输的可靠性
  • tcp只能一对一通信，udp支持一对一、一对多、多对多通信。
  • tcp适合要求可靠传输的场景，例如文件传输；udp适合实时应用。

简述tcp三次握手和四次挥手的过程

三次握手：

  1. 第一次握手：客户端向服务端发送 SYN 请求报文段（设置一个随机序列号seq=x），表示希望建立连接

  2. 第二次握手：服务器收到SYN报文段之后，回应一个确认报文，并设置 SYN（seq=y） 和 ACK标志位（ACK标志位的序列号为ack=x+1）报文给客户端

  3. 第三次握手：客户端收到服务端的确认报文后，回应一个 ACK（ack=y+1） 报文

为什么是3次，不是4次或2次？

答：因为TCP连接的双方都是全双工的，全双工是指既要发送又要接受数据。这就要求TCP协议要确认双方的收发数据的能力。2次握手验证不全，3次握手能完成的事没必要多做一次，所以既不是2次也不是4次。

四次挥手：

  1. 第一次挥手，客户端向服务端请求断开连接
  2. 第二次挥手，服务端向客户端发出确认报文，同时进入等待关闭状态，通知上层应用连接即将关闭，作好相应准备
  3. 第三次挥手，服务器将剩余数据发送完后，服务端向客户端发送FIN报文，请求断开连接
  4. 第四次挥手客户端向服务端发出确认报文，客户端进入TIME-WAIT（时间等待）状态，防止服务端未收到确认报文而无法进入关闭状态，等待一段时间（2*MSL（最长报文段寿命））后进入关闭状态，服务端在接收到客户端的确认报文后也进入关闭状态，随后连接断开。

为什么第四次挥手后客户端没有立刻进入关闭状态，而是等待了一段时间？

答：防止确认报文丢失，服务端无法正常关闭。如果报文丢失，服务端一段时间后会再次发送请求断开连接的FIN报文，如果客户端没有等待直接关闭，就收不到这个，导致服务端无法正常关闭。

浏览器渲染过程，重排重绘

  • 浏览器的绘制流程：
  1. 解析HTML，构建DOM树
  2. 解析CSS，构建CSSOM树
  3. 将DOM和CSSOM合并成渲染树
  4. 根据渲染树来布局，计算每个节点的尺寸和位置
  5. 调用GPU绘制在屏幕上

  • 重排/回流：

    重新确定DOM节点的尺寸和位置。当DOM的尺寸、位置发生变化或者增删时，就会触发重排。

    比如当width、padding、margin、left、top等属性更改，就会触发重排。

  • 重绘：

    重新计算样式树，并绘制在屏幕上。

    当页面的样式发生改变时，就会触发重绘。比如当color、background-color、visibility等属性更改时，就会触发重绘。

    重排一定会触发重绘，重绘不一定触发重排。

  • 如何减少重排、重绘操作

    重排、重绘操作会阻塞主线程，当耗时过长时会造成页面卡顿。通过以下方式减少重排、重绘：

    • 读写 offset族、scroll族和client族属性的时候，会强制队列（大多数浏览器都会通过队列优化批量修改的场景）刷新，触发重排操作。

      不要重复获取这类属性，最好缓存起来。

    • 多用transform实现动画，transform不会触发重排重绘

    • 使用cssText或css class优化批量修改css操作。

    • 离线修改DOM。批量修改DOM前，先通过display: none让元素脱离文档流，再批量修改DOM，最后让元素回到文档流。

    • 现代浏览器已经通过队列优化批量修改的场景，cssText、离线DOM修改主要是针对老版本的优化。

    • 用途：用于存储临数据例，例如表单数据

  • localStorage：本地存储

    • 存储期限：存储在磁盘中，除非用户主动删除，否则数据会一直存在。
    • 存储容量：约5MB
    • 用途：用于持久化存储数据，例如用户信息、主题等

  • cookies：通过key-value格式存储数据的小型文本文件。可以设置过期时间，如果不设置，在会话结束后被删除

    • 存储期限：可以设置过期时间，如果不设置，会话结束时被删除。
    • 存储容量：约4kb，每个域名最多存储20个cookie
    • 用途：用于存储少量的，随请求一起发送给服务器的数据，例如sessionid

  • IndexedDB

    非关系型数据库，数据存储在磁盘中。存储容量很大，可以存储数百mb或者更多。支持存储对象、数据等结构化数据。适合存储大量数据的场景。

因为localStorage支持数据持久化，除非手动删除否则不会清除。

  • 当需要存储临时数据，例如表单时，使用sessionStorage；

什么是跨域，跨域中的options是什么？

  • 什么是跨域？

    一个url由协议、域名、端口三个部分组成：http://localhost:8080。当一个请求的url和当前页面的url有任意一个部分不同，就是跨域。

    默认情况下浏览器不允许发送跨域请求，在服务端设置相应的响应头来解决跨域问题。

  • 为什么会存在跨域

    因为浏览器的同源策略，默认情况下只允许访问协议域名端口相同的url，目的是提高安全性，防止xss等跨站请求攻击。

  • options请求：预检请求

    预检请求是浏览器在执行某些跨域请求之前，先发送的一种特殊的 HTTP 请求，目的是询问服务端是否允许跨域，以及允许哪些请求方法、请求头等。

  • options请求过多带来的问题

    主要是性能问题，大量的options请求会增加网络延迟和服务器负担。

  • 如何减少options请求？

    • 尽量使用“简单请求”
    • 限制自定义请求头的使用

  • 什么是简单请求？

    请求方法为GET、POST或HEAD。且只包含特定的几个请求头，并且不包含自定义请求头

  • 跨域的表现是什么，请求无响应还是根本没发？

    请求正常发送，服务器正常响应，只是被浏览器拦截了，控制台会提示你违反了跨域策略。

路由懒加载

对于SPA项目，项目打包后，会把所有页面打包成一个文件，JS文件会很大。用户打开网站会一次性加载所有资源，首屏加载会很慢。

修改webpack、vite等打包工具的配置实现代码分块，配合路由懒加载，即可实现页面资源的懒加载。

  • 懒加载的前提：ES6的动态加载模块——import()

组件懒加载

某些组件例如弹窗组件不是一进入页面就展示，而是需要用户手动触发才显示出来，这种情况可以使用组件懒加载。

和路由懒加载的目的一样，都是为了资源拆分，减少JS文件的体积来提高加载速度。

适用场景：

  • 组件不是刚进入页面就加载，并且页面的JS文件体积很大，影响加载速度。
  • 组件复用性高，很多页面都有引入，利用组件懒加载抽离出该组件，可以更好地利用缓存。

注意，有时资源拆分的过细也不好，会造成浏览器http请求过多。

JS的加载方式

  • 普通脚本：浏览器遇到<script>标签，会同步加载和执行JS脚本，阻塞dom的解析和渲染。
  • async模式：异步加载脚本，不会阻塞dom解析。加载完会立即执行脚本（执行是同步的，会阻塞dom解析）
  • defer模式：异步加载脚本，不会阻塞dom解析。在文档被解析、渲染后执行，适合依赖dom的脚本。
  • type: module：表示ES模块，ES6引入。行为等同于defer，如果设置了async属性，那脚本在加载完会立即执行。（vite会把所有非ES模块编译为ES模块）

图片优化

如果一个网页的图片过多，会导致加载速度变慢，解决方式：

  • 图片压缩：原图和压缩后的图片体积差距非常大，但是效果差距不大。通过裁剪和压缩图片降低图片体积，提高加载速度。
  • 图片懒加载。许多图片可能在页面的底部，未出现在视口中，不需要一开始全部加载完。

webpack 和 vite 的其他区别

  • 开发阶段的构建方式

    webpack项目中，每次启动开发服务器，需要重新构建、打包整个应用，项目规模越大速度越慢；

    vite基于原生ES模块的动态加载机制，在开发阶段直接跳过了构建打包（需要预构建），只有当模块被使用到，浏览器发送请求后，vite才会把对应源码编译为ES模块，提供给浏览器，相当于实现了模块懒加载。大大提高了项目的构建速度（和热更新速度）。

  • 热更新：

    webpack项目中，每次修改文件，都会对整个项目重新进行打包，这对大项目来说，是非常不友好的。

    vite监听到文件变更后，会用websocket通知浏览器哪些模块有变化，浏览器重新发起新的请求，只替换发生变更的模块。

  • 学习和使用成本：

    我们使用webpack自己去搭建项目脚手架时，需要配置比较多的东西， 比如：跨域、代码压缩、代码分割、css预处理器的代码转换、样式兼容性、vue/react代码解析、图片压缩、代码热更新、es降级、ts转换等等，使用成本很高。

    vite对我们常用功能都做了内置，vite官方也提供了一些官方模板、社区模板，我们可以快速地创建项目，不需要关心太多的配置项。学习成本低。

  • 编程语言：

    webpack是基于nodejs编写的，但js只能单线程运行，无法利用多核CPU的优势；

    vite开发阶段的编译（预构建与按需编译），都是使用esbuild完成的。esbuild是用go语言编写的，可以充分利用多核CPU的优势。

mvvm将项目分为model层、view层、viewmodel三层

git

Git 中的工作区、暂存区、本地仓库和远程仓库

  • **工作区：**就是计算机里的项目文件夹
  • **暂存区：**一个临时存储区域，用来存储需要提交到本地仓库的文件。通过git add命令将工作区的文件添加到暂存区
  • **本地仓库：**储存本地分支和远端分支，通过git commit命令可以将暂存区的修改合并到本地仓库
  • **远程仓库：**部署在远程服务器的仓库。将代码push到远程仓库前，如果此前有别人提交新的commit，需要先进行合并（通过git fetch+git merge或git pull命令）

git pull 和 git fetch + git merge 的区别

  • git fetch：获取远程仓库的最新更新，并更新到本地的远程分支仓库，本地分支不会更新。
  • git pull：获取远程分支的更新，更新到本地的远程分支仓库，然后自动将本地分支和远程分支进行合并。

所以git pull = git fetch + git merge。

git rebase的作用

把本地由分支合并导致的分叉提交历史整理成直线，目的是让提交历史更简洁，容易查看。

git常见命令

  • git add：将文件添加到暂存区
  • git commit：将暂存区的修改提交到本地仓库
  • git fetch：获取远程分支的最新更新，更新到本地的远端仓库
  • git merge：将本地分支和远程分支合并
  • git pull = git fetch + git merge
  • git rebase：变基，作用是将由于分支合并导致的分叉提交历史整理成一条直线，目的是让提交历史更简洁，容易查看。、
  • git branch：创建分支
  • git checkout/git switch：切换分支
  • git checkout -b/git s：创建并切换分支

多人协作时怎么管理分支？

  • 首先创建两个分支：master分支作为生产分支，dev分支作为开发分支。并且不在这两个分支上进行开发
  • 当开发一个新功能，或者修复一个bug时，从dev切出一个新的feature或者bugFix分支，功能完成后合并到dev分支并删除
  • 发布新版本时，将dev分支合并到master分支。
